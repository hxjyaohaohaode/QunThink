# 分布式系统架构详细设计

## 1. 架构演进策略

### 1.1 当前状态分析
当前系统为单体Node.js应用：
- 单进程Express服务器
- 内存WebSocket连接
- 本地SQLite数据库
- 单点故障风险

### 1.2 目标架构
微服务分布式架构：
- 服务拆分（聊天、社交、文件、AI、记忆）
- 水平扩展能力
- 高可用性设计
- 数据一致性保证

### 1.3 演进路径
**阶段1：服务拆分** - 将单体拆分为独立服务
**阶段2：数据分离** - 数据库分库分表
**阶段3：负载均衡** - 引入负载均衡器和服务发现
**阶段4：高可用** - 多副本部署和故障转移

## 2. 微服务拆分设计

### 2.1 服务定义

#### 2.1.1 聊天服务 (chat-service)
**职责**：
- 消息路由和广播
- WebSocket连接管理
- 会话状态维护
- 房间/群组管理

**接口**：
```protobuf
service ChatService {
  rpc SendMessage(MessageRequest) returns (MessageResponse);
  rpc GetMessages(GetMessagesRequest) returns (MessageList);
  rpc JoinGroup(JoinRequest) returns (JoinResponse);
  rpc LeaveGroup(LeaveRequest) returns (LeaveResponse);
  rpc GetOnlineUsers(GroupRequest) returns (UserList);
}
```

#### 2.1.2 社交互动服务 (social-service)
**职责**：
- 智能点赞和评论处理
- 社交行为分析
- 互动质量评估
- 社交关系管理

**接口**：
```protobuf
service SocialService {
  rpc AutoLike(AutoLikeRequest) returns (LikeResponse);
  rpc AddComment(CommentRequest) returns (CommentResponse);
  rpc GetSocialStats(StatsRequest) returns (SocialStats);
  rpc AnalyzeInteraction(InteractionRequest) returns (AnalysisResult);
}
```

#### 2.1.3 文件处理服务 (file-service)
**职责**：
- 文件上传和存储
- 内容解析和提取
- 格式转换
- 文件元数据管理

**接口**：
```protobuf
service FileService {
  rpc UploadFile(stream FileChunk) returns (UploadResponse);
  rpc ParseFile(ParseRequest) returns (ParseResponse);
  rpc GetFileInfo(FileRequest) returns (FileInfo);
  rpc SearchFiles(SearchRequest) returns (FileList);
}
```

#### 2.1.4 AI推理服务 (ai-service)
**职责**：
- 多AI模型调用
- 负载均衡和故障转移
- 结果聚合和缓存
- 模型性能监控

**接口**：
```protobuf
service AIService {
  rpc GenerateResponse(AIRequest) returns (AIResponse);
  rpc GetAIModels(Empty) returns (ModelList);
  rpc GetModelStats(ModelRequest) returns (ModelStats);
  rpc SwitchModel(SwitchRequest) returns (SwitchResponse);
}
```

#### 2.1.5 记忆服务 (memory-service)
**职责**：
- 长期记忆存储
- 语义检索
- 记忆关联
- 引用跟踪

**接口**：
```protobuf
service MemoryService {
  rpc StoreMemory(MemoryRequest) returns (MemoryResponse);
  rpc RetrieveMemory(RetrieveRequest) returns (MemoryList);
  rpc LinkMemories(LinkRequest) returns (LinkResponse);
  rpc SearchMemories(SearchRequest) returns (MemoryList);
}
```

### 2.2 服务通信设计

#### 2.2.1 同步通信（REST/gRPC）
- 服务间直接调用使用gRPC
- 对外API使用REST/GraphQL
- 请求超时设置：默认3秒
- 重试策略：指数退避，最多3次

#### 2.2.2 异步通信（消息队列）
- 使用Redis Streams或Apache Kafka
- 事件驱动架构
- 最终一致性保证
- 死信队列处理

#### 2.2.3 WebSocket通信
- 聊天服务负责WebSocket连接
- 其他服务通过事件通知聊天服务
- 消息广播使用Redis Pub/Sub

## 3. 数据存储设计

### 3.1 数据库分片策略

#### 3.1.1 垂直分片
```
用户相关数据 → PostgreSQL Cluster 1
  - 用户信息
  - 群组信息
  - 社交关系

消息相关数据 → PostgreSQL Cluster 2
  - 消息内容
  - 评论回复
  - 互动记录

文件相关数据 → PostgreSQL Cluster 3
  - 文件元数据
  - 解析内容
  - 存储位置
```

#### 3.1.2 水平分片
**分片键**：用户ID哈希
**分片数量**：初始4个分片，支持动态扩展
**分片算法**：一致性哈希

### 3.2 缓存策略

#### 3.2.1 Redis集群配置
```
主节点：3个（读写分离）
从节点：6个（读负载均衡）
哨兵节点：3个（故障检测和转移）
```

#### 3.2.2 缓存数据分类
```
1. 会话缓存（TTL: 24小时）
   - 用户会话
   - WebSocket连接状态
   - 临时数据

2. 热点数据缓存（TTL: 1小时）
   - 热门群组消息
   - 活跃用户信息
   - AI模型配置

3. 计算结果缓存（TTL: 30分钟）
   - 情感分析结果
   - 相关性评分
   - 社交行为分析
```

### 3.3 文件存储设计

#### 3.3.1 存储分层
```
热存储（SSD）：
  - 最近上传的文件（30天内）
  - 高访问频率文件
  - 小文件（<10MB）

温存储（HDD）：
  - 历史文件（30-90天）
  - 中等访问频率文件

冷存储（对象存储）：
  - 归档文件（>90天）
  - 低访问频率文件
  - 备份数据
```

#### 3.3.2 文件分区
```
分区策略：按时间分区
目录结构：/uploads/{year}/{month}/{day}/{file_id}
备份策略：跨区域复制
```

## 4. 负载均衡设计

### 4.1 全局负载均衡（DNS级）
```
DNS轮询 → CDN边缘节点 → 区域负载均衡器 → 应用负载均衡器
```

### 4.2 应用负载均衡（Nginx/HAProxy）
```
配置：
- 轮询 + 权重分配
- 最少连接数算法
- 健康检查（每10秒）
- SSL终止
- 请求缓冲
```

### 4.3 服务发现（Consul/Etcd）
```
服务注册：
  1. 服务启动时向注册中心注册
  2. 定期发送心跳（每30秒）
  3. 服务下线时注销

服务发现：
  1. 客户端查询注册中心
  2. 获取可用服务列表
  3. 负载均衡选择实例
  4. 缓存结果（TTL: 60秒）
```

## 5. 高可用设计

### 5.1 多副本部署
```
每个服务至少3个副本
部署在不同可用区
使用Kubernetes Deployment管理
```

### 5.2 故障转移
```
检测机制：
  - 健康检查端点
  - 心跳检测
  - 性能监控

转移策略：
  - 自动重启（最多3次）
  - 服务重新调度
  - 流量重定向
```

### 5.3 数据备份和恢复
```
实时备份：
  - 数据库主从复制
  - 文件系统快照
  - 日志归档

恢复策略：
  - 最近恢复点（RPO < 5分钟）
  - 恢复时间目标（RTO < 15分钟）
```

## 6. 监控和告警

### 6.1 监控指标
#### 6.1.1 基础设施监控
```
- CPU使用率（阈值: 80%）
- 内存使用率（阈值: 85%）
- 磁盘使用率（阈值: 90%）
- 网络带宽（阈值: 70%）
```

#### 6.1.2 应用监控
```
- 请求延迟（P95 < 500ms）
- 错误率（< 0.5%）
- 吞吐量（QPS/TPS）
- 服务可用性（> 99.9%）
```

#### 6.1.3 业务监控
```
- 活跃用户数
- 消息发送量
- AI响应时间
- 互动频率
```

### 6.2 告警策略
```
等级1（紧急）：
  - 服务完全不可用
  - 数据丢失风险
  - 安全漏洞

等级2（重要）：
  - 性能严重下降
  - 部分功能失效
  - 资源严重不足

等级3（警告）：
  - 轻微性能问题
  - 资源使用率高
  - 异常模式检测
```

## 8. 部署架构

### 8.1 Kubernetes集群配置
```
控制平面：3个节点（高可用）
工作节点：初始6个节点（可扩展）
节点规格：4核CPU，16GB内存，100GB SSD
```

### 8.2 服务部署配置
```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: chat-service
spec:
  replicas: 3
  selector:
    matchLabels:
      app: chat-service
  template:
    metadata:
      labels:
        app: chat-service
    spec:
      containers:
      - name: chat-service
        image: chat-service:latest
        ports:
        - containerPort: 8080
        resources:
          requests:
            memory: "512Mi"
            cpu: "250m"
          limits:
            memory: "1Gi"
            cpu: "500m"
        livenessProbe:
          httpGet:
            path: /health
            port: 8080
          initialDelaySeconds: 30
          periodSeconds: 10
        readinessProbe:
          httpGet:
            path: /ready
            port: 8080
          initialDelaySeconds: 5
          periodSeconds: 5
```

### 8.3 自动扩缩容（HPA）
```yaml
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: chat-service-hpa
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: chat-service
  minReplicas: 3
  maxReplicas: 10
  metrics:
  - type: Resource
    resource:
      name: cpu
      target:
        type: Utilization
        averageUtilization: 70
  - type: Resource
    resource:
      name: memory
      target:
        type: Utilization
        averageUtilization: 80
```

## 11. 风险评估和缓解

### 11.1 技术风险
**风险**：分布式系统复杂性导致调试困难
**缓解**：完善的日志和追踪系统，使用OpenTelemetry

**风险**：数据一致性问题
**缓解**：采用最终一致性模型，实现补偿事务

**风险**：服务间通信延迟
**缓解**：连接池优化，请求合并，缓存策略

### 11.2 运维风险
**风险**：部署复杂性增加
**缓解**：完善的CI/CD流水线，蓝绿部署策略

**风险**：监控盲点
**缓解**：多层次监控，从基础设施到业务指标

**风险**：容量规划不足
**缓解**：自动扩缩容，定期性能测试

### 11.3 业务风险
**风险**：用户体验下降
**缓解**：渐进式迁移，A/B测试，用户反馈收集

**风险**：成本增加
**缓解**：资源优化，自动扩缩容，预留实例使用
