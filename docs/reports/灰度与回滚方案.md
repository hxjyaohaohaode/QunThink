# 灰度与回滚方案

## 灰度发布

### 发布前检查

- 后端测试通过：`cd backend && npm test`
- 前端构建通过：`cd frontend && npm run build`
- 契约文档通过：`node scripts/validate-openapi.mjs`
- 安全审计通过：CI 中 `npm audit --audit-level=high`

### 灰度步骤

1. 先部署后端，再部署前端，避免旧前端访问新接口时缺少后端能力。
2. 首批只开放给内部账号或测试账号，重点验证：
   - 登录/注册
   - 群创建
   - 消息发送/删除
   - 点赞点踩
   - 文件上传/下载/分析
3. 观察运行日志、上传目录、WebSocket 连接状态与错误率。
4. 验证通过后再扩大到全部用户。

## 回滚策略

### 触发条件

- 登录或消息主链路出现阻断
- 文件上传成功但无法下载或分析
- 前端构建产物导致页面白屏
- 新接口契约导致关键页面不可用

### 回滚步骤

1. 回滚前端到上一个已验证构建版本。
2. 回滚后端到上一个已验证发布版本。
3. 保持用户数据目录不做 destructive 处理，禁止删除 `backend/data`。
4. 重新执行健康检查与冒烟验证：
   - `GET /api/health`
   - 登录
   - 获取群组
   - 发送消息
   - 上传文件

## 数据与配置注意事项

- 本轮未引入破坏性数据迁移，主要是契约收敛与工程化补齐。
- 群创建字段以 `ai_members` 为唯一标准，回滚时要确保旧前端不会再次发送 `aiMembers`。
- 文件真实归属仍以 `req.userId` 为准，回滚时不得恢复对客户端 `uploader_id` 的信任。

## 推荐发布节奏

- 开发环境验证
- 预发布环境回归
- 小流量灰度
- 全量发布

## 建议保留物

- 上一个可用前端构建包
- 上一个可用后端镜像或压缩包
- 当前 `backend/data` 目录备份
- 本次 `docs/reports/一次性修复总览报告.md`
