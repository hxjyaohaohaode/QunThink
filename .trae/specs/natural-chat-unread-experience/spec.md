# 群聊自然交互与未读消息体验优化 Spec

## Why
当前AI群聊存在三个核心体验问题：(1) AI发言有明显的顺序感，像排队发言而非真人自然交互，且AI经常复制他人发言内容；(2) AI之间的点赞/评论互动不够智能，评论内容是硬编码的固定话术，缺乏上下文感知；(3) 用户浏览历史消息时新消息会打断阅读，缺少微信风格的未读消息提示和精准跳转。

## What Changes
- **BREAKING**: 重构发言调度引擎，移除基于index的顺序延迟，改为完全独立的随机延迟+意愿评分机制，让AI"想说就说，不想说就不说"
- 增强AI system prompt，强化"禁止复制他人发言"规则，要求AI必须用自己的语言重新表达
- 重构AI评论生成逻辑，从硬编码话术改为基于上下文的AI生成评论
- 优化点赞/评论的视觉反馈，确保互动信息出现在对应消息的正确位置
- 实现进入群聊时自动滚动到最新消息
- 实现微信风格未读消息提示：右下角浮动徽章显示未读数量，点击跳转到第一条未读消息（非底部）

## Impact
- Affected specs: 发言调度引擎、AI system prompt、社交互动服务、消息列表组件、未读消息组件
- Affected code:
  - `backend/src/services/scheduler/index.js` - 发言调度核心逻辑
  - `backend/src/services/ai/index.js` - AI system prompt
  - `backend/src/services/social/index.js` - 社交互动服务
  - `frontend/src/components/Chat/MessageList.tsx` - 消息列表与滚动逻辑
  - `frontend/src/components/Chat/NewMessageBadge.tsx` - 未读消息提示
  - `frontend/src/components/Chat/MessageBubble.tsx` - 消息气泡（点赞/评论展示）

## ADDED Requirements

### Requirement: AI自然发言调度
系统应让AI像真人一样在群聊中自然发言，而非按顺序排队。

#### Scenario: AI独立决策是否发言
- **WHEN** 用户发送一条消息
- **THEN** 每个AI独立计算发言意愿分
- **AND** 意愿分低于阈值的AI保持沉默，不发送任何消息
- **AND** 意愿分高的AI发言，但彼此之间没有固定顺序
- **AND** 各AI的发言延迟完全独立随机计算，不依赖其他AI的index

#### Scenario: AI发言时机自然分布
- **WHEN** 多个AI同时决定发言
- **THEN** 各AI的"正在输入"状态独立出现
- **AND** 发言完成时间有先有后，呈现自然交错
- **AND** 不出现所有AI同时开始输入、同时完成的情况

#### Scenario: AI选择沉默
- **WHEN** 某个AI的发言意愿分低于阈值
- **THEN** 该AI不显示"正在输入"状态
- **AND** 该AI不发送任何消息
- **AND** 其他AI的发言不受该AI沉默的影响

### Requirement: AI禁止复制他人发言
系统应确保AI不会复制粘贴其他成员的原话。

#### Scenario: AI引用他人观点
- **WHEN** AI同意或回应其他成员的观点
- **THEN** AI必须用自己的风格和语言重新表达
- **AND** 不得出现与前面发言高度相似的段落
- **AND** 可以引用关键词但必须加入自己的分析和见解

### Requirement: AI智能评论生成
系统应让AI之间的评论互动基于上下文智能生成，而非使用固定话术。

#### Scenario: AI对他人消息发表评论
- **WHEN** AI决定对某条消息发表评论
- **THEN** 评论内容基于该消息的上下文和AI自身人设生成
- **AND** 评论内容与消息主题相关，不是泛泛的固定话术
- **AND** 评论通过WebSocket实时推送到前端

#### Scenario: AI对他人消息点赞
- **WHEN** AI决定对某条消息点赞
- **THEN** 点赞信息实时通过WebSocket推送到前端
- **AND** 消息气泡上立即显示点赞数增加和点赞者标识

### Requirement: 进入群聊自动滚动到最新消息
系统应在用户进入群聊时自动定位到最新消息。

#### Scenario: 用户切换到某个群聊
- **WHEN** 用户点击进入某个聊天群
- **THEN** 消息列表自动滚动到最底部显示最新消息
- **AND** 滚动完成后将状态标记为"已在底部"

#### Scenario: 首次打开应用进入默认群聊
- **WHEN** 用户首次打开应用并进入默认群聊
- **THEN** 消息列表自动滚动到最底部显示最新消息

### Requirement: 微信风格未读消息提示
系统应在用户浏览历史消息时，以微信风格提示未读新消息。

#### Scenario: 用户在浏览历史消息时有新消息到达
- **WHEN** AI发送新消息且用户不在消息列表底部
- **THEN** 在消息列表右下角显示一个浮动提示徽章
- **AND** 徽章显示未读消息数量（如"3条新消息"）
- **AND** 消息列表不自动跳转到底部，保持用户当前阅读位置

#### Scenario: 用户点击未读消息提示
- **WHEN** 用户点击右下角的未读消息提示徽章
- **THEN** 消息列表平滑滚动到第一条未读消息的位置
- **AND** 第一条未读消息短暂高亮显示
- **AND** 如果还有更多未读消息，徽章更新为剩余未读数量
- **AND** 如果已到达底部，徽章消失

#### Scenario: 用户手动滚动到底部
- **WHEN** 用户手动将消息列表滚动到底部附近
- **THEN** 未读消息提示徽章自动消失
- **AND** 未读计数重置为0

#### Scenario: 用户滚动到第一条未读消息后仍有未读
- **WHEN** 用户点击徽章跳转到第一条未读消息
- **AND** 第一条未读消息之后还有更多未读消息
- **THEN** 徽章继续显示剩余未读数量
- **AND** 用户可再次点击徽章跳转到下一条未读消息

## MODIFIED Requirements

### Requirement: 发言调度引擎
原有调度引擎使用index-based顺序延迟，现修改为完全独立的随机延迟。

**原逻辑**：
- `calculateInitialDelay` 使用 `index * 500` 产生顺序延迟
- `selectParticipatingAIs` 阈值较低（30分），大部分AI都会发言
- 所有参与AI同时显示"正在输入"状态

**新逻辑**：
- 移除 `index * 500`，改为每个AI独立计算随机延迟（1~8秒范围）
- 提高参与阈值，让更多AI选择沉默（意愿分>55才发言，40~55有50%概率，<40沉默）
- "正在输入"状态在AI实际开始生成前才显示（延迟结束后），而非一开始就全部显示

### Requirement: AI评论生成
原有评论使用硬编码话术数组，现修改为基于上下文的智能生成。

**原逻辑**：
- 评论内容从固定数组随机选取：`['同意这个观点！', '这个角度很独特', ...]`

**新逻辑**：
- 评论内容基于目标消息的上下文和AI人设，调用AI API生成简短评论
- 如果API调用失败，降级为基于消息关键词的模板评论（而非完全随机）

### Requirement: 未读消息提示组件
原有NewMessageBadge组件点击后直接滚动到底部，现修改为微信风格精准跳转。

**原逻辑**：
- 点击徽章 → `scrollToBottom()` 滚动到最底部
- 未读计数仅跟踪数量，不跟踪消息位置

**新逻辑**：
- 点击徽章 → 滚动到第一条未读消息的位置
- 跟踪第一条未读消息的ID，用于精准定位
- 滚动后更新未读计数（减去已跳过的消息数）
