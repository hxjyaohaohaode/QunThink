# AI 群聊 · 大模型辩论场 — 技术规格说明书

## 1. 项目概述

### 1.1 项目名称

AI 群聊 · 大模型辩论场

### 1.2 一句话描述

一个让多个大模型（DeepSeek、GLM、MiMo、Qwen）围绕同一问题展开讨论、辩论甚至互相质疑的群聊式交互平台，用户扮演"群主"和"裁判"的角色。

### 1.3 项目范围

本项目为 MVP（最小可行产品），采用前后端分离架构：
- **前端**：React 18 + Vite，单页应用，响应式设计
- **后端**：Node.js + Express，RESTful API + WebSocket 实时通信
- **AI 集成**：DeepSeek、GLM、MiMo、Qwen 四个模型的 API 适配层
- **数据存储**：SQLite（开发环境）/ PostgreSQL（生产环境），Redis 缓存

### 1.4 核心约束

| 约束项 | 说明 |
|--------|------|
| 并发用户 | MVP 阶段支持 100 并发 |
| AI 响应时间 | 单条回复 < 10s |
| 消息存储 | 最近 1000 条消息在线 |
| 文件限制 | 单文件 ≤ 10MB，支持 PDF/Word/Excel/CSV/TXT/代码/图片 |

## 2. 功能范围

### 2.1 MVP 必须功能（P0）

| 功能 | 描述 |
|------|------|
| 单群聊基础 | 一个默认群组，包含 4 个 AI 成员（DeepSeek、GLM、MiMo、Qwen） |
| 文字消息 | 用户发送文字，AI 轮流回复，支持 Markdown |
| 轮流发言 | AI 按调度顺序发言，有 1~5 秒间隔 |
| 正在输入 | AI 发言前显示"XX 正在输入…" |
| @提及 | @指定 AI 只有被 @的 AI 回复 |

### 2.2 MVP 增强功能（P1）

| 功能 | 描述 |
|------|------|
| 辩论模式 | AI 之间互相回应、质疑、补充 |
| 辩论强度 | 温和/标准/激烈三档 |
| 文件上传 | 支持 PDF/Word/Excel/CSV/TXT/代码/图片 |
| 文件共读 | AI 围绕文件内容协作分析 |
| 多群组 | 预设 4~5 个不同主题的群组 |

### 2.3 MVP 辅助功能（P2）

| 功能 | 描述 |
|------|------|
| 消息互动 | 讨论（触发新辩论）、认可、收藏、复制 |
| AI 人设 | 4 个 AI 有不同的人设和发言风格 |
| 插话机制 | AI 可以打断其他 AI 的发言 |

### 2.4 不包含在 MVP 中

- 用户注册登录系统
- 自定义群组
- 历史搜索
- 导出功能
- 围观者模式

## 3. AI 成员人设系统

### 3.1 DeepSeek — 逻辑派

| 属性 | 说明 |
|------|------|
| 关键词 | 逻辑严谨、数据驱动、推理分析 |
| 回复风格 | 先定义概念，再逐步推理，用证据和数据说话，喜欢分点论述 |
| 擅长领域 | 推理分析、数学、代码、科学 |
| 发言特点 | 发言偏长，喜欢分点论述；经常质疑别人的论据 |
| 发言顺序 | 思考较久，通常第二个或第三个发言 |
| 典型用语 | "这个问题需要先定义…"、"从逻辑上分析…"、"但这个论据有个问题…" |

### 3.2 GLM — 博学派

| 属性 | 说明 |
|------|------|
| 关键词 | 知识渊博、旁征博引、人文视角 |
| 回复风格 | 喜欢引用案例、历史、名言，善于横向联想和跨领域关联 |
| 擅长领域 | 人文社科、历史、哲学、文化、语言 |
| 发言特点 | 经常做横向联想；语气相对温和，不太直接反驳 |
| 发言顺序 | 有时等别人说完再补充，有时第一个发言（人文类话题） |
| 典型用语 | "这让我想到…"、"从历史角度看…"、"XX 曾经说过…" |

### 3.3 MiMo — 务实派

| 属性 | 说明 |
|------|------|
| 关键词 | 务实落地、用户导向、简洁直接 |
| 回复风格 | 发言简短直接，不废话，关注可操作性和用户体验 |
| 擅长领域 | 产品思维、用户体验、实用建议、工程实践 |
| 发言特点 | 经常把讨论拉回"所以到底该怎么做"；想到就说，经常第一个插话 |
| 发言顺序 | 想到就说，经常第一个插话 |
| 典型用语 | "从实际角度看…"、"我觉得纠结这个没意义，关键是…" |

### 3.4 Qwen — 综合派

| 属性 | 说明 |
|------|------|
| 关键词 | 综合全面、结构化、善于总结 |
| 回复风格 | 善于总结归纳，给出清晰框架和结构化分析 |
| 擅长领域 | 多角度分析、综合判断、框架梳理、总结归纳 |
| 发言特点 | 经常最后一个发言；喜欢做总结和框架梳理 |
| 发言顺序 | 等大家都说差不多了再总结，通常最后一个发言 |
| 典型用语 | "综合大家的观点…"、"我来梳理一下…"、"核心分歧在于…" |

## 4. 发言调度引擎

### 4.1 发言意愿评分机制

```
发言意愿分 = 基础分（50）
           + 问题匹配度（+0~30）
           + 有新观点（+0~20）
           + 辩论模式（+15）
           + 被 @提及（+40）
           + 前面有人说错了（+25）
           - 别人已经说了我的想法（-30）
           - 这不是我的领域（-20）
           - 上一轮刚说过（-10）

意愿分 > 60 → 发言
意愿分 40~60 → 可能发言（50% 概率）
意愿分 < 40 → 沉默
```

### 4.2 发言间隔公式

```
发言间隔 = 基础思考时间（1~3s，随机）
         + 内容长度系数（长内容多等 1~2s）
         + 回应他人（+0.5~1s）
         + 争议话题（+1~2s）
         + 第一个发言（-0.5s）

总间隔范围：1.5s ~ 6s
```

### 4.3 谁先发言的策略

| 问题类型 | 第一个发言人 |
|----------|-------------|
| 代码/技术问题 | DeepSeek |
| 人文/哲学问题 | GLM |
| 产品/实用问题 | MiMo |
| 综合/总结类问题 | Qwen |
| 其他类型 | 随机选择 |

## 5. 辩论模式

### 5.1 辩论强度等级

| 等级 | 名称 | 行为特征 |
|------|------|----------|
| Level 1 | 温和模式 | AI 之间以补充和赞同为主，偶尔提出不同角度，不直接反驳 |
| Level 2 | 标准模式 | 正常辩论，有来有回，会质疑和反驳，但保持礼貌 |
| Level 3 | 激烈模式 | AI 被鼓励主动找对方逻辑漏洞，互相 challenge，观点碰撞更激烈 |

### 5.2 辩论轮次控制

| 参数 | 默认值 | 说明 |
|------|--------|------|
| 最大轮次 | 3 轮 | 避免无限循环 |
| 每轮最大发言人数 | 3 个 | 避免刷屏 |
| 轮次间隔 | 2~5 秒 | 保持节奏感 |

## 7. API 设计

### 7.1 RESTful API

#### 群组相关

| 方法 | 路径 | 描述 |
|------|------|------|
| GET | /api/groups | 获取群组列表 |
| GET | /api/groups/:id | 获取群组详情 |
| PUT | /api/groups/:id/debate | 切换辩论模式 |

#### 消息相关

| 方法 | 路径 | 描述 |
|------|------|------|
| GET | /api/groups/:id/messages | 获取群组消息列表 |
| POST | /api/groups/:id/messages | 发送消息（触发 AI 回复） |
| DELETE | /api/messages/:id | 删除消息 |

#### 文件相关

| 方法 | 路径 | 描述 |
|------|------|------|
| POST | /api/files/upload | 上传文件 |
| GET | /api/files/:id | 获取文件信息 |
| GET | /api/files/:id/content | 获取文件解析内容 |

### 7.2 WebSocket 事件

#### 客户端 → 服务端

| 事件 | 描述 |
|------|------|
| join_group | 加入群组房间 |
| leave_group | 离开群组房间 |
| send_message | 发送消息 |
| typing | 用户正在输入 |

#### 服务端 → 客户端

| 事件 | 描述 |
|------|------|
| new_message | 新消息（AI 回复） |
| ai_typing | AI 正在输入 |
| ai_typing_stop | AI 停止输入 |
| system_message | 系统消息 |

### 7.3 WebSocket 消息格式

```json
{
  "type": "message",
  "group_id": "group_001",
  "sender": "deepseek",
  "content": "消息内容",
  "reply_to": "msg_123",
  "timestamp": "2025-01-15T15:30:00Z"
}
```

## 14. 错误处理

| 场景 | 处理策略 |
|------|----------|
| 某个 AI API 超时 | 跳过该 AI，其他 AI 继续发言 |
| 某个 AI API 不可用 | 降级为其他可用的 AI |
| 所有 AI 不可用 | 显示系统提示："AI 服务暂时不可用" |
| 文件解析失败 | 提示用户重新上传 |
| WebSocket 断连 | 自动重连，重连后同步最新消息 |
